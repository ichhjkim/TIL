# 최적화의 원리



## 옵티마이저와 실행 계획



### 옵티마이저

개발자가 SQL을 작성, 실행할때 옵티마이저는 SQL을 어떻게 실행할 것인지를 계획

SQL 실행 계획 수립, 관리 시스템 소프트웽ㅓ

옵티마이저는 SQL  성능에 아주 중요한 역할



### 특징

데이터 딕셔너리에 있는 오브젝트 통계, 시스템 통계등의 정보를 사용해서 예상되는 비용 산정

옵티마이저는 여러 계획중 최저 비용을 가지고 있는 계획 선택해 SQL 실행



### 옵티마이저 실행 계획 

> 옵티마이저가 비효율적으로 실행 계획을 수립하면 SQL개발자는 SQL을 개선해야한다. 이때 옵티마이저에게 실행계획을 변경할 수 있도록 요청할 수 있는데 힌트(HINT)를 사용



옵티마이저는 SQL실행계획을 `PLAN_TABLE`에 저장

그래서 개발자는 `PLAN_TABLE`을 조회해서 실행 계획을 확인할 수 있따. 



TOAD -> Execution Plan Current SQL



## 옵티마이저 종류

`개발자 SQL` -> `Parsing(SQL 문법검사/구문분석)` -> `규칙기반 옵티마이저`/`비용기반 옵티마이저` (Data Dictionary)-> `실행계획(PLAN_TABLE) ` -> SQL 실행 -> 데이터 인출(Fetch)



### 옵티마이저 엔진

- Query Transformer : SQL을 효율적으로 실행하기 위해서 옵티마이저가 변환, SQL이 변환되어도 결과가 동일
- Estimator: 통계 정보를 활용해서 `SQL 실행 비용`을 계산, 총비용은 최적의 실행계획을 수립하기 위해서
- Plan Generator: SQL을 실행할 실행 계획을 수립



#### 비용기반 옵티마이저

오브젝트 통계 및 시스템 통계를 사용해서 총 비용 계산 

총비용이라는 것은 SQL문을 실행하기 위해서 예상되는 소요시간 혹은 자원의 사용량을 의미

단, 비용기반 옵티마이저에서 통계정보가 부적절한 경우 성능저하가 발생할 수 있다



## 인덱스

인덱스는 데이터를 빠르게 검색할 수 있는 방법을 제공

인덱스는 인덱스 키로 정렬되어 있다. -> 원하는 정보를 빠르게 조회

오름차/내림차순으로 탐색 가능

하나의 테이블에 여러개의 인덱스 생성 가능, 하나의 인덱스는 여러개의 칼럼으로 구성 가능

기본키는 자동으로 인덱스가 만들어 지고 이름은 STSXXXX이다.



### 구조

`Root` Block -> `Branch` Block -> `Leaf` Block



- ROOT BLOCK : 다음 단계의 주소를 가지고 있는 포인터
- LEAF BLOCK: 인덱스 키와 ROW ID로 구성되고 인덱스 키는 정렬되어 저장 // Double Linked LIst 형태로 되어 있어서 양방향 탐색 가능/ LEAF BLOCK에서 인덱스키를 읽으면 ROWID를 사용해서 테이블의 행을 직접 읽을 수 있다



### 생성

CREATE INDEX <인덱스 이름> <테이블 이름> ON <테이블 이름>  (컬럼이름 정렬기준, 컬럼이름2,정렬기준, ...)



### 인덱스 스캔



- 인덱스 유일 스캔

  - 인덱스의 키값이 중복되지 않는 경우, 해당 인덱스를 사용할 때 발생

- 인덱스 범위 스캔

  - 특정 범위를 조회하는 WHERE문을 사용할 경우 발생(LIKE, BETWEEN)

  - INDEX RANGE SCAN은 LEAF BLOCK의 특정범위를 스캔한 것

- 인덱스 전체 스캔

  - 인덱스에서 검색되는 인덱스 키가 많은 경우 LEAF BLOCK의 처음부터 끝까지 전체를 읽는다.



## 옵티마이저 조인



### `Nested Loop` 조인

하나의 테이블에서 데이터를 먼저 찾고 그 다음 테이블을 조인

먼저 조회되는 테이블 == 외부 테이블

그 다음 조회되는 테이블 == 내부 테이블

#### 특징

외부테이블의 크기가 작은 것 을 먼저 찾아야함

random access가 많이 됨 -> 이걸 줄여야 성능 향상



### `Sort Merge` 조인

두개의 테이블은 `SORT_AREA`라는 메모리 공간에 모두 로딩하고 SORT 수행

그다음 병합(MERGE)



#### 특징

- SORT가 발생하기 때문에 데이터 양이 많아지면 성능이 떨어짐
  - 정렬 데이터 양이 많아지면 임시영역에서 수행. 임시영역은 디스크에 있기 떄문에 성능이 급격히 떨어짐



##### 힌트

`/*+ ordered use_merge(b) */*`



-> use_merge 힌트를 사용해서 SORT MERGE 조인 가능, 단 use_merge는 ordered 힌트와 같이 사용해야함



### HASH 조인

- 두개의 테이블 중 작은 테이블은 `HASH메모리`에 로딩, 두개의 테이블의 조인키를 사용해서 해시테이블을 생성
- 해쉬 조인은 해시함수를 사용해서 주소를 계산하고 해당 주소를 사용해서 테이블을 조인해서 CPU연산이 많다.

- HASH 조인 시에는 선행 테이블이 충분이 메모리에 로딩되는 크기여야 한다.
