# 데이터 모델링



## 데이터모델링의 이해

> 현실세계를 데이터베이스로 표현하기 위해 추상화
>
> - 고객의 비즈니스 프로세스 이해
> - 비즈니스 프로세스의 규칙을 이해
> - 규칙을 데이터 모델로 표현



### 특징

| 특징   | 설명                              |
| ------ | --------------------------------- |
| 추상화 | 현실세계를 간략화                 |
| 단순화 | 누구나 쉽게 이해할 수 있도록 표현 |
| 명확성 | 명확하게 의미 해석 + 한가지 의미  |



### 데이터 모델링을 할 때 유의할 사항

| 종류     | 설명                                                         |
| -------- | ------------------------------------------------------------ |
| 중복     | 데이터베이스가 여러 장소에 같은 정보를 저장하지 않도록       |
| 비유연성 | 데이터의 정의를 데이터의 사용 프로세스와 분리함으로써 데이터 모델링은 데이터 혹은 프로세스의 작은 변화가 어플리케이션과 데이터 베이스에 중대한 변화를 일으킬 수 있는 가능성을 줄인다 |
| 비일관성 | 데이터와 데이터 간의 상호연관관계에 대해 명확하게 정의해 명확하게 정의.<br />사용자가 처리하는 프로세스 혹은 이와 관련된 프로그래과 테이블의 연계성을 높이는 것은 데이터모델의 업무변경에 대해 취약하게 만드는 단점 |



### 데이터 모델링 단계

| 단계   | 설명                                                         |
| ------ | ------------------------------------------------------------ |
| 개념화 | - 추상화 수준 높<br />- 업무 중심적<br />- 포괄적인 수준<br />- 전사적 데이터 모델링<br />- EA 수립시 많이 이용 |
| 논리적 | - Key<br />- 속성<br />- 관계등을 정확히 표현<br />- 재사용성이 높음 |
| 물리적 | 실제로 데이터 베이스에 이식할 수 있도록 성능, 저장등 물리적인 성격을 고려(성능, 보안, 가용성등) |



### 데이터모델링 관점

| 관점              | 설명                                                         |
| ----------------- | ------------------------------------------------------------ |
| 데이터            | 비즈니스 프로세스에서 사용되는 데이터<br />- 구조분석/정적분석 |
| 프로세스          | 비즈니스 프로세스에서 수행하는 작업<br />- 동적/시나리오 분석<br /> |
| 데이터와 프로세스 | 데이터와 프로세스관의 관계<br />- CRUD                       |



### ERD 작성 순서

1. 엔티티 그리기
2. 엔티티 배치(중요 엔티티 왼쪽 상단)
3. 엔티티간 관계 설정-- 
4. 엔터티 관계명 기술-- 
5. 관계의 `참여도` 기술--
6. 관계의 `필수`여부



#### ERD 작성시 고려 사항

- 중요 엔터티 가급적 왼쪽 상단
- ERD는 이해가 쉬워야 하고 복잡하지 않아야함



### 데이터모델링 고려사항

- 독립성: 정규화로 데이터 중복 제거
- 고객 요구사항의 표현
- 데이터 품질 확보 : 데이터 표준을 정의, 표준준수율 관리



## 3층 스키마

> 사용자, 설계자, 개발자가 데이터 베이스를 보는 관점에 따라 데이터베이스를 기술하고 이들 간의 관계를 정의한 ANSI 표준
>
> - 목적
>   - 독립성 확보: 데이터복잡도 증가/중복제거/사용자요구사항 반영등의 장점
> - 3층 스키마의 독립성
>
> | 독립성        | 설명                                                         |
> | ------------- | ------------------------------------------------------------ |
> | 논리적 독립성 | 저장구조가 변경되어도 응용프로그램 및 개념 스키마에 영향 없다 |
> | 물리적 독립성 | 데이터 베이스 논리적 구조가 변경되어도 응용프로그램에 영향이 없다 |



### 구조

| 단계        | 설명                                                         |
| ----------- | ------------------------------------------------------------ |
| 외부스키마  | - 사용자 관점<br />- 업무상 관련있는 데이터 접근<br />- 관련 데이터베이스의 `뷰(VIEW)` 표시<br />- `응용프로그램`이 접근하는 데이터 베이스 정의 |
| 개념스키마  | - 설계자 관점<br />- 사용자 전체집단의 데이터베이스 구조<br />- 전체 DB 내 규칙/구조 표현<br />- `통합`데이터베이스 구조 |
| 내부 스키마 | - 개발자 관점<br />- 데이터 베이스의 `물리적` 저장 구조<br />- 데이터 저장구조/레코드 구조/필드 정의/ 인덱스 등을 의미 |



## 엔터티

업무에서 관리애햐하는 데이터 집합/

개념/사건/장소 등의 명사



### 특징

- 식별자: 엔터티는 유일한 식별자가 있어야 한다.
- 인스턴스 집합: 2개 이상
- 속성 : 반드시 속성을 가지고 있어야 한다.(2개 이상)
- 관계; 다른 엔터티와 최소 1개이상의 관계가 있어야 함
- 업무: 업무에서 관리되어야 하는 집합



### 종류



#### 유형과 무형에 따른 종류

| 종류        | 설명                                                         |
| ----------- | ------------------------------------------------------------ |
| 유형 엔터티 | 업무에서 도출되며 지속적으로 사용<br />- 고객, 강사, 사원등<br />- 사람같이 물리적 형태 |
| 개념 엔터티 | - 물리적 형태가 없다<br />- 개념적으로 사용되는 엔터티       |
| 사건 엔터티 | - 비즈니스 프로세스를 실행하면서 생성                        |



#### 발생 시점에 따른 엔터티 종류

| 종류        | 설명                                                         |
| ----------- | ------------------------------------------------------------ |
| 기본 엔터티 | 키 엔터티<br />- 다른 엔터티에 영향 받지 않고, 독립적으로 생성<br />- 예) 고객, 상품, 부서 |
| 중심 엔터티 | 기본 엔터티와 행위 엔터티간 중간<br />- 기본 엔터티로부터 발생 -> 행위엔터티 생성<br />예) 계좌, 주문, 취소 |
| 행위 엔터티 | 2개 이상의 엔터티로부터 발생<br />예) 주문 이력              |





## 속성

업무에서 필요한 정보인 엔터티가 가지는 항목

더이상 분리되지 않는 단위, 업무에 필요한 데이터를 저장

인스턴스의 구성요소/`의미적으로 더이상 분해되지 않는다`.



### 특징

- `하나의 값`만 가짐
- 주식별자에게 함수적으로 종속
- 기본키가 변경되면 속성의 값이 변함



### 종류



#### 분해여부에 따른 속성의 종류

| 종류    | 설명                                                         |
| ------- | ------------------------------------------------------------ |
| 단일    | 하나의 의미                                                  |
| 복합    | 여러개의 의미 예) 주소                                       |
| 다중 값 | 여러개의 값을 가질 수 있는 것<br />- 다중값 속성은 엔터티로 분리됨 |



#### 특성에 따른 속성의 종류

| 종류 | 설명                                                         |
| ---- | ------------------------------------------------------------ |
| 기본 | 비즈니스 프로세스에서 도출 되는 본래의 속성<br />- 예) 회원ID, 이름, 계좌번호 |
| 설계 | 데이터모델링 과정에서 발생되는 속성- 유일한 값을 부여<br />예) 상품코드, 지점코드 |
| 파생 | 다른 속성에 의해 만들어지는 속성<br />예) 합계, 평균         |



## 관계

엔티티간의 관련성을 의미

존재관계와 행위관계로 분류



### 종류

- 존재관계: 엔터티간 상태를 의미
- 행위관계: 엔터티간에 어떤 행위가 있는 것



### 관계 차수

2개의 엔티티간 관계에 참여하는 수



### 종류

- 1:1 관계
  - 완전 1:1 : 하나의 엔티티에 관계되는 엔티티 관계가 반드시 하나 있는 경우
  - 선택적 1:1: 하나의 엔터티에 관계되는 엔터티의 관계가 하나이거나 없을 수도 있다
- 1:N
- M:N
  - M:N은 카테시안의 곱이 있을 수 있어서 해소시켜야함
- 필수적 관계(|)와 선택적 관계(O)
  - 필수적 관계: 반드시 하나가 있어야 하는 관계
  - 선택적 관계: 없을 수도 있는 관계 



### 식별관계와 비식별관계



- 식별관계(PK)
  - 독립적으로 존재가능한 `강한 개체`
  - 다른 엔터티와 관계를 가질때 기본키를 공유
- 비식별관계(FK)
  - 강한개체의 기본키를 다른에터티의 기본키가 아닌 일반칼럼으로 관계를 가진다(FK)



## 엔터티 식별자

- 주식별자: 기본키

  - 유일성
  - 최소성
  - 대표

  만족



| 키     | 설명                                         |
| ------ | -------------------------------------------- |
| 기본키 | 유일성, 최소성, 대표 만족                    |
| 후보키 | 유일성, 최소성 만족                          |
| 슈퍼키 | 유일성 만족/ 최소성 만족 하지 않음           |
| 대체키 | 여러개의 후보키 중 기본키를 선정하고 남은 키 |



### 식별자의 종류



#### 대표성 여부에 따른 식별자 종류

- 주식별자: 유일성, 최소성을 만족, 다른엔터티와 참조관계로 연결(기본키)
- 보조식별자: 유일성과 최소성은 만족하지만 대표성을 만족하지 못함(후보키/대체키)



#### 생성 여부

- 내부식별자: 엔터티내부에서 스스로 생성
- 외부식별자: 다른 엔터티와 관계로 인해 만들어짐(FK)



#### 속성의 수에 따른 식별자 종류

- 단일 식별자: 하나의 속성으로 구성
- 복합 식별자: 2개이상의 속성



#### 대체여부

- 본질 식별자: 비즈니스 프로세스에서 만들어지는 식별자
- 인조 식별자: 인위적으로 만들어지는 식별자



# 데이터 모델과 성능



## 정규화

일관성, 최소한의 데이터 중복, 최대한의 데이터 유연성을 위한 방법 

제1-5정규화까지 있지만 실질적으로 제3정규화까지만 수행



### 절차

| 정규화절차 | 설명                                                         |
| ---------- | ------------------------------------------------------------ |
| 제1정규화  | - 속성의 `원자성 `확보<br />- **기본키**를 잡는 것           |
| 제2정규화  | 기본키가 **2개 이상**인 경우 부분함수 종속성을 제거          |
| 제3정규화  | - 기본키를 제외한 `칼럼간 종속성 제거`<br />- `이행함수 종속성 제거` |
| BCNF       | 기본키를 제외, 후보키가 있는 경우, 후보키가 기본키를 종속시키면 분해 |
| 제4정규화  | 여러 칼럼들이 하나의 칼럼을 종속시키는 경우 분해/`다중값 속성을 제거` |
| 제5정규화  | 조인에 의해 종속성 발생시 분해                               |



## 정규화화 성능

데이터 중복 제거 -> 데이터의 유연성을 높임

입출력 데이터의 양을 줄여서 성능을 향상

한행을 읽기 위해 DISK IO를 줄여줌

### 정규화의 문제점

- JOIN 유발
- CPU와 메모리를 많이 사용



-> 이 문제점을 해결하기 위해 반정규화



## 반정규화

데이터 베이스 성능 향상을 위해 

데이터 중복 허용, 조인을 줄임



### 특징

- 조회 속도 향상(SELECT)
- 유연성은 낮아짐



### 반정규화를 수행하는 경우

- 수행속도가 느려지는 경우
- 다량의 범위를 자주 처리
- 특정 범위의 데이터만 자주 처리
- 요약/집계 정보가 자주 요구되는 경우



#### 클러스터링

인젝스 정보를 저장할 때 물리적으로 정렬해서 저장

조회시에 인접블록을 연속적으로 읽기 때문에 성능 향상



### 반정규화 절차

대상 조사 및 검토 -> 다른 방법 검토 -> 반정규화 수행



### 반정규화기법

- 계산된 칼럼 추가
- 테이블 수직 분할: 컬럼 기준으로 분할
- 데이블 수평분할: 값을 기준으로 분할
- 테이블 병합
  - 단, 1:N 테이블을 병함하면 많은 양의 데이터 중복 발생



#### 파티션 테이블의 장점

- 데이터 조회시 Access 범위가 줄어서 성능 향상
- 데이터 분할되어 있어 `IO 성능 향상`
- 각 파티션을 독립적으로 `백업 및 복구` 가능



### 슈퍼타입 및 서브타입 변환방법

| 변환방법      | 설명                                                         |
| ------------- | ------------------------------------------------------------ |
| OnetoOne Type | 수퍼타입과 서브타입을 개별테이블로 도출<br />- 테이블 수가 많아서 조인 많이 발생 |
| Plus TYPE     | 슈퍼타입과 서브타입 테이블로 도출<br />- 조인 많이 발생      |
| Single TYPE   | - 수퍼타입+서브타입 -> 하나의 테이블<br />- 조인 성능 굳 IO 성능 BAD |



## 분산 데이터 베이스

물리적으로 떨어진 데이터 베이스를 네트워크로 연결하여 단일 데이터 베이스 이미지를 보여주고 분산된 작업처리를 수행하는 DB



### 투명성 종류

- 분할
- 위치
- 지역사상
- 중복
- 장애 및 병행 투명성



### 장점과 단점
